01 Heap = Priority Queue

The core claim of this lecture
A heap and a priority queue are the same data structure. Whether someone says "heap" or "priority queue," they mean the same thing.

A heap/priority queue is a particular type of binary tree — specifically one where the elements are organized so the "most important" one (smallest, for a min heap, or largest, for a max heap) is always instantly accessible at the root.

The "priority queue" framing: imagine people waiting in line, but instead of first-come-first-served, whoever has the highest priority (lowest number, in this lecture's convention) gets served first — celebrities and VIPs skip the line. A min heap processes items in exactly that order: smallest value first.

02 Array Representation

Before even talking about the heap property, any binary tree can be represented as an array. Root at index 0, and for any index i:

leftChild(i)  = 2·i + 1
rightChild(i) = 2·i + 2

Important distinction made early in the lecture: just representing a binary tree as an array does not make it a heap. A tree only becomes a heap once it satisfies the heap property (parent ≤ children for a min heap) — turning an arbitrary tree/array into one is done via a function called heapify.

03 Min Heap & Heap Pop (Extract-Min)

Once the array above is run through minHeapify, it becomes a valid min heap: the root is the minimum, and every node is smaller than both of its children (recursively).

Heap Pop — worked trace

Remove root
Take -4 out (the current minimum) — this is what gets returned.
Promote smaller child
Of the root's two children, promote the smaller one upward. Between the two candidates, the value 0 is chosen over 1 because 0 < 1.
Repeat down the path
The gap left behind gets filled by promoting the smaller of its children — continue this down a single path until you reach a node with only one child (or none), then promote whatever's left.
Why O(log n)
Heap pop only ever fixes one path down the tree — never branches out to fix multiple paths. As long as the tree stays roughly height-balanced (which a complete tree guarantees), that path has length log₂ n. So extract-min is O(log n).

"Heap pop" and "extract min" refer to the same operation — remove and return the root, then repair the tree along one path.

04 Heap Push (Insert)

Also called heap push. Insert makes space for the new value at the next open array slot, then repeatedly checks "are we happy with its position?" — swapping upward until the answer is yes.

Worked trace — inserting 7

Place
Make space at the next open slot for 7.
Check & swap
"Are we happy with its position?" — No: 7 is smaller than its parent 9, so it belongs above. Swap.
Check again
Now compare 7 against its new parent 2. 7 is bigger than 2, so we are happy — stop.

Worked trace — inserting 13

Place & check
13 is placed, compared to its parent — it's bigger, which is exactly what a min heap wants below a smaller parent. Already happy — no swap needed.

Worked trace — inserting -2

Place & check
-2 is smaller than its parent 10 — not happy, swap.
Keep climbing
Still smaller than the next parent up — swap again. In the worst case this climbs all the way to the root.
Why O(log n)
Same reasoning as heap pop: insertion only ever climbs one branch of the tree upward. Each step up effectively halves how much of the tree is left to consider — repeatedly dividing n by 2 — giving a worst case of O(log₂ n).

05 Heap Peek

"Heap peek" just means asking what the minimum currently is, without removing it. Since the minimum is always at the root — array index 0 — this is a direct index lookup.

06 Heap Sort

The whole algorithm
Repeatedly pop the minimum off the heap and append it to a results list.

Worked trace: popping -2 first, then re-fixing the tree (promote 0, then 1, then 10 settles into place) — then popping 0, re-fixing again — then popping 1, and so on. Each pop is O(log n), and there are n elements to pop.

MetricValueWhy
TimeO(n log n)n pops, each O(log n) to repair the tree
Space (as taught/coded here)O(n)Building a new output array
Space (theoretical minimum)O(1)Possible via in-place swapping — noted as correct but "rather complex," not implemented in this lecture
Why heap sort is considered excellent
O(n log n) is among the best achievable time complexities for comparison-based sorting — matching merge sort — while heap sort can additionally be done with O(1) extra space in its optimized in-place form (unlike merge sort, which needs O(n)).

07 Heapify — Build Heap in O(n)

heapify is the function that takes an arbitrary binary tree/array and rearranges it in place to satisfy the heap property. The process: start from the last non-leaf node (leaves are skipped — a single node is trivially a valid heap) and sift down each one into its correct position, working from the bottom of the tree up to the root.

Sift down — what it means

For a given node, compare it against its children; if it's not the smallest of the three, swap with whichever child is smallest, then continue checking from that new position further down.

Worked trace (matching the lecture's example)

Start at last non-leaf
Ignore all leaves. Start with the node 2 (rightmost non-leaf). Its children include 1 — the smallest of the three — so swap 2 and 1.
Next non-leaf: 10
Smallest of its three values is -4 → swap.
Next: 3
3 is smaller than 7 (good) but bigger than 2 (bad) → swap with 2.
Next: 5
5 ties with another 5 and is smaller than 8 — already in a valid position, no swap needed (a no-op swap would change nothing).
Next: -7 and -10
Both are already smaller than their children — no swaps needed.
Root: 12
Needs the most sifting — swaps down repeatedly, promoting the smallest child at each level, until -10 lands at the root.

Final check: every node from leaves up to root is smaller than its children (or tied), and -10 sits at the very top — a valid min heap.

Why O(n), not O(n log n)
A naive count (n/2 non-leaf calls × O(log n) each) overestimates. Most non-leaf nodes are near the bottom of the tree, where sift-down only travels a short distance — the total work sums to a small constant multiple of n. The lecture states the result directly (O(n)) without deriving the full proof, calling it "an extremely profound part of computer science."
MetricValue
Time complexityO(n)
Space complexityO(1) — rearranges the array in place

08 Max Heap

Everything above mirrors exactly for a max heap — every node is bigger than its two children instead of smaller, and the maximum sits at the root. Extracting (heap pop / extract-max) promotes the larger child at each step instead of the smaller.

Example from the lecture: root 5, children 3 and 2; 3's children 1 and 0; 2's children -1 and -5. Every parent is ≥ both of its children — valid max heap.

09 Storing Tuples (Priority + Data)

A detail the lecture calls out as commonly glossed over: in real use (and in LeetCode problems), you rarely put bare numbers on a heap — you store (priority, data) tuples so the heap sorts by priority while carrying the actual payload along with it.

Pattern
Put the value you want to sort/compare by first in the tuple (e.g. priority number), and the associated data second. Example: (1, "person A"), (2, "person B") — the heap orders by the first element, and ties break by comparing the second.

Popping then gives you back both the priority and the data associated with it — a "true" priority queue where the thing you retrieve is meaningful application data, not just a bare number.

10 Python heapq — Full Toolkit

Python's built-in heapq module only implements a min heap — there is no built-in max-heap mode. The lecture works entirely with plain arrays/lists as the underlying heap storage.

import heapq

# --- heapify: O(n), rearranges an existing array in place ---
a = [2, 5, 0, 1, 10, 8, -4, 3, 12, 9]
heapq.heapify(a)  # -4 becomes the root; rest of array is NOT fully sorted

# --- heap push: O(log n), insert a new value ---
heapq.heappush(a, 4)

# --- heap pop: O(log n), remove & return the minimum ---
minimum = heapq.heappop(a)

# --- heap push+pop combined: O(log n) total (faster than calling both separately) ---
result = heapq.heappushpop(a, 99)  # pushes 99, then pops & returns the new minimum

# --- peek at minimum: O(1), no library call needed ---
smallest = a[0]

# --- size: O(1) ---
size = len(a)

Heap Sort using heapq

def heap_sort(array):
    heapq.heapify(array)          # O(n)
    n = len(array)
    new_list = [0] * n
    for i in range(n):
        new_list[i] = heapq.heappop(array)  # O(log n) each, O(n log n) total
    return new_list

Simulating a Max Heap (negate everything)

Since heapq is always a min heap, the trick to get max-heap behavior is to negate every value before heapifying — the smallest negative number corresponds to the largest original number.

b = [2, 5, 0, 1, 10, 8, -4, 3, 12, 9]
n = len(b)
for i in range(n):
    b[i] = -b[i]           # negate everything first
heapq.heapify(b)            # now behaves like a max heap on the original values

largest = -heapq.heappop(b) # negate back on the way out

# inserting 7 into this "max heap" means pushing its negation:
heapq.heappush(b, -7)   # this represents inserting +7

Building a heap incrementally (vs. heapify)

c = [-5, 4, 2, 1, 7, 0, 3]
heap = []
for x in c:
    heapq.heappush(heap, x)   # O(log n) each time, O(n log n) total
Important distinction
Pushing n items one at a time is O(n log n) — noticeably slower than calling heapq.heapify() once on the full array, which is O(n). Prefer heapify whenever you already have all the data upfront.

Tuples on the heap (frequency example)

from collections import Counter

d = [5, 5, 5, 5, 4, 4, 4, 3, 3]
counter = Counter(d)   # {5: 4, 4: 3, 3: 2}

heap = []
for key, value in counter.items():
    heapq.heappush(heap, (value, key))  # (frequency, number) — sorts by frequency first

# smallest frequency now sits at heap[0]

12 Mistakes & Interview Tips

❌ Assuming an array is a heap just because it "looks tree-shaped"
Any binary tree can be written as an array via the index formulas — that alone does not make it satisfy the heap property. Only after heapify is run does the parent/child ordering actually hold.
❌ Assuming heapq.heapify() produces a fully sorted array
It only guarantees the heap property (root is min, parent ≤ children) — the rest of the array is not in ascending order. Don't index into it expecting sorted values beyond position 0.
❌ Pushing elements one at a time instead of heapifying
If you already have the full array, call heapify() (O(n)) instead of looping with heappush() n times (O(n log n)).
❌ Forgetting Python's heapq has no native max-heap mode
You must negate values going in and negate again coming out — a common LeetCode gotcha.
💡 Tuple trick for real interview problems
Whenever you need to track "priority + payload" (e.g. frequency + value, distance + point, task time + task id), push a tuple (priority, payload). The heap sorts by the first tuple element automatically, with the second breaking ties.
