01 Heap vs Binary Search Tree

A heap is a binary tree data structure that stores items and maintains an ordering between them — similar to a BST, but with a much more relaxed rule.

Binary Search TreeHeap
Ordering rule Left subtree < parent ≤ right subtree (applies recursively, globally) Parent vs. its direct children only (min heap: parent ≤ children)
Strictness Strict — full ordering across the whole tree Relaxed — only a parent/child relationship, no rule between siblings
Consequence More constraints → costlier to maintain Fewer constraints → cheaper/faster to maintain
Key Insight
A heap's ordering is more relaxed than a search tree's ordering. Less strict rules → easier to maintain → faster code (better Big-O for insert/remove).

Important: there is no relationship between siblings. Swapping two entire sibling subtrees still produces a valid heap.

02 Min Heap vs Max Heap

Min Heap
Parent ≤ children (recursively).
The root is always the true minimum of the whole set — it has no parent, so nothing can be smaller than it.
Max Heap
Parent ≥ children (recursively).
The root is always the true maximum of the whole set.

03 Complete Binary Tree

A heap is always implemented as a complete binary tree — this is what keeps it balanced (short height), which is where the efficiency comes from.

Definition
Every level is completely filled except possibly the last level, and the nodes on that last level are pushed as far left as possible. (Structure only — values don't matter here.)

To keep a tree complete while inserting: always fill nodes left to right, and only start a new level once the current one is full.

Why completeness matters

A complete tree with n nodes has height O(log n) — dramatically shorter than a degenerate (linked-list-like) tree with the same node count. Since heap operations walk up/down the height of the tree, this log-height is exactly what makes heap insert/remove logarithmic.

04 Array Representation

Heaps are not implemented with node/pointer objects like a typical tree. Because the tree is always complete, we can store it in a plain array and use index math to find parents/children — no pointers needed.

Why this works
Because the tree is always complete, pushing to the end of the array is always equivalent to adding a new node at the next left-most open slot — structurally identical to normal tree insertion, but O(1) to place.

05 Insertion — Sift Up

Insertion has two steps:

  1. Push the new value onto the end of the array (= bottom level, left-most open slot → keeps tree complete).
  2. Sift up: while the new node is smaller than its parent (min heap), swap it with its parent. Repeat until it's in the correct spot or reaches the root.

Worked trace — inserting 4 into an existing min heap

Heap before insert: 7, 11, 10, 8 (as a tree, 7 is root, 11 & 10 are its children, 8 is 11's child).

Step 1 — Place
Push 4 to the end of the array (bottom-left open slot). Tree stays complete, but heap order is broken: 4 < 6 (its parent).
Step 2 — Compare
4 < parent → swap. Compare again with the new parent → still smaller → swap again. Continue until 4 becomes the new root, or its parent is ≤ 4.
Step 3 — Stop
Stop sifting when the node is either at the root (index 0) or no longer smaller than its parent.
Rule
Sift up while child < parent (min heap) — swap and move up. For a max heap, sift up while child > parent.

Complexity

StepTimeWhy
Push to array endO(1)Array append
Sift upO(log n)Worst case travels the full height of a complete tree = log n
Total TimeO(log n)
SpaceO(1)Just a loop + a few index variables (excluding the O(n) already used to store the heap)

06 Extraction — Sift Down (Extract-Min)

extractMin() removes and returns the root (the minimum, for a min heap). Steps:

  1. Save the root value (this is what gets returned).
  2. Move the last element of the array into the root position (keeps the tree complete — no gaps).
  3. Sift down: while the new root is greater than one of its children, swap with the smaller of the two children. Repeat.
Key Insight
When choosing which child to swap with, always pick the smaller child (min heap) — swapping with the larger child can leave the other side out of order. A missing child is treated as +Infinity so the comparison logic never needs special-casing "no left/right child".

Worked trace

1. Save root
Root = 5 (index 0) → this will be returned at the end.
2. Move last → root
Pop the last array element and place it at index 0. Tree is complete again, but likely out of order.
3. Sift down
Compare new root to left = 2i+1 and right = 2i+2. If root > smaller child → swap with that child, move down, repeat.
4. Stop
Stop when the node is ≤ both children (or it has no children — treated as ≤ Infinity).

Complexity

StepTimeWhy
Save root + move last elementO(1)Direct index access + array pop
Sift downO(log n)Worst case travels full tree height = log n
Total TimeO(log n)
SpaceO(1)Constant extra variables

07 JavaScript MinHeap Implementation

Full array-backed min heap, matching the sift up / sift down logic above.

class MinHeap {
  constructor() {
    this.array = [];
  }

  isEmpty() { return this.array.length === 0; }
  size()    { return this.array.length; }

  // --- INSERT: push to end, then sift up ---
  insert(value) {
    this.array.push(value);
    this.siftUp(this.array.length - 1);
  }

  siftUp(idx) {
    let currentIndex = idx;
    while (currentIndex > 0) {
      const parentIndex = Math.floor((currentIndex - 1) / 2);
      if (this.array[currentIndex] < this.array[parentIndex]) {
        this.swap(currentIndex, parentIndex);
        currentIndex = parentIndex;
      } else {
        break; // correctly positioned
      }
    }
  }

  // --- EXTRACT MIN: swap root with last, pop, sift down ---
  extractMin() {
    if (this.isEmpty()) return null;
    if (this.size() === 1) return this.array.pop();

    const min = this.array[0];
    this.array[0] = this.array.pop(); // last element becomes new root
    this.siftDown(0);
    return min;
  }

  siftDown(startIdx) {
    let currentIndex = startIdx;
    while (currentIndex < this.array.length - 1) {
      const leftIdx  = 2 * currentIndex + 1;
      const rightIdx = 2 * currentIndex + 2;

      const leftVal  = this.array[leftIdx]  === undefined ? Infinity : this.array[leftIdx];
      const rightVal = this.array[rightIdx] === undefined ? Infinity : this.array[rightIdx];

      const smallerIdx = leftVal < rightVal ? leftIdx : rightIdx;
      const smallerVal = leftVal < rightVal ? leftVal : rightVal;

      if (this.array[currentIndex] > smallerVal) {
        this.swap(currentIndex, smallerIdx);
        currentIndex = smallerIdx;
      } else {
        break; // correctly positioned
      }
    }
  }

  swap(i, j) {
    [this.array[i], this.array[j]] = [this.array[j], this.array[i]];
  }
}
Max Heap variant
Flip every comparison: siftUp while child > parent; siftDown swaps with the larger child; missing children default to −Infinity instead of +Infinity.
Note
JavaScript has no built-in heap data structure. In interviews, it's usually fine (and expected) to assume a working heap API (insert, extractMin/extractMax) and reason about complexity — you're rarely expected to hand-roll one from scratch under time pressure.

08 Pattern: Top K Elements (using a Heap)

Classic use case: find the Kth largest, K smallest, most frequent, etc. A heap lets you avoid fully sorting the input — you only need it partially ordered.

8.1 — Kth Largest Element

Trick: use a MIN heap of size K (not a max heap). Insert every number; whenever the heap grows past size K, extract-min (throw away the smallest). After processing everything, the heap's root is the Kth largest.

Why a min heap?
The heap keeps only the K largest values seen so far. Each time it overflows past K, the smallest of that group is discarded — because it can no longer be among the K largest. What survives at the end are exactly the K largest values, and the root (heap minimum) of that group is the Kth largest overall.
function findKthLargest(nums, k) {
  const heap = new MinHeap();
  for (const num of nums) {
    heap.insert(num);
    if (heap.size() > k) heap.extractMin(); // discard smallest of the surplus
  }
  return heap.extractMin(); // root == Kth largest
}

8.2 — K Smallest Elements

Mirror image: use a MAX heap of size K. Insert everything; when the heap exceeds size K, extract-max (throw away the largest of the surplus). What remains are the K smallest — reverse the extraction order to return them ascending.

function kSmallest(nums, k) {
  const heap = new MaxHeap();
  for (const num of nums) {
    heap.insert(num);
    if (heap.size() > k) heap.extractMax(); // discard largest of the surplus
  }
  const result = [];
  while (!heap.isEmpty()) result.push(heap.extractMax());
  return result.reverse(); // was descending, flip to ascending
}
Pattern to remember
"Kth largest / top-K largest" → min heap capped at size K (discard the small stuff).
"Kth smallest / bottom-K smallest" → max heap capped at size K (discard the big stuff).
You always keep a heap of the opposite extreme so the thing you throw away is the thing you don't need.

Sorting vs. Heap — complexity comparison

ApproachTimeSpaceNotes
Full sort, then indexO(n log n)O(1) in-place / O(n) if copyingSimple, but sorts everything even though you only need K values
Heap of size KO(n log K)O(K)Only partially orders the data — faster since K ≤ n, often K ≪ n

The n comes from iterating every element once; the log K comes from insert/extract on a heap that never grows past size K.

10 Mistakes & Interview Tips

❌ Forgetting to keep the tree complete
Always insert at the array's end / remove by swapping in the array's last element. Never leave "holes" — that's what keeps the O(log n) height guarantee.
❌ Swapping with the wrong child during sift down
Min heap: always swap with the smaller of the two children, not just "a child that's smaller than the parent" — picking the wrong one can leave the tree out of order on the other side.
❌ Not handling missing children
Treat an out-of-bounds child index as +Infinity (min heap) or −Infinity (max heap) so your comparison code doesn't need special-case branches.
❌ Using max heap when you need Kth largest (or vice versa)
Counter-intuitive but important: Kth largest uses a min heap (bounded at size K), because you're discarding the small values that can't be in the top K.
💡 Interview framing
You're rarely asked to implement a heap from scratch — assume a working heap API and focus on explaining why a heap (partial ordering) beats full sorting for Top-K style problems, and state the O(n log K) complexity clearly.
