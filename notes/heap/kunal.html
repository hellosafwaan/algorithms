01 Why Heaps?

Start from the problem, not the data structure. You're given a stream of n items one at a time, and your real requirement is: at any point, instantly (ideally O(1)) hand back the item that matches some condition — the smallest, the largest, whatever "priority" means for the problem.

Two naive approaches — and why both fail

ApproachGet min/maxInsert new itemProblem
Unsorted arrayO(n) — linear scanO(1) — just appendQuery is slow
Keep array sorted at all timesO(1) — it's at index 0O(n log n) — must re-sort on every insertInsert is slow
The gap heaps fill
You want O(1) query for the min/max and a cheap insert. Full sorting is overkill — you don't need the whole array ordered, you only need the top item guaranteed correct. A heap gives you O(1) peek and O(log n) insert by keeping just enough order (parent vs. children) instead of full order (every pair).

Example use case from the lecture: given IMDb-style review scores as they stream in, always be able to report the current highest score instantly. That's exactly the shape of problem a heap solves.

02 Heap Structure

A heap is stored internally as an array, but visualized/reasoned about as a complete binary tree — every level full except possibly the last, and the last level filled left to right.

Convention used in this series
The array is 1-indexed — index 0 is left empty/unused, and the root lives at index 1. This is a deliberate choice because it makes the parent/child formulas clean.
left(i)   = 2·i
right(i) = 2·i + 1
parent(i) = ⌊i / 2⌋

No node objects, no .left / .right pointers needed — the formulas alone locate any parent or child directly from an index.

Max Heap property

Every node's value is greater than or equal to all of its children (recursively). This guarantees the maximum is always at the root — but says nothing about the ordering of the rest of the array. The array does not need to be sorted; only the parent≥children rule matters.

03 Insertion — Up-Heap

Insert always goes to the next open left-most slot (end of array), then bubbles upward while it violates the heap property. This lecture calls this operation up-heap.

Worked trace — building a min heap by inserting 8, 7, 9, 10, 11, 12, 14, 13, 5

Insert 8, 7
7 < parent(8) → swap. Root becomes 7.
Insert 9…14
Each new value is already ≥ its parent along the insertion path, so no swaps needed — array grows: 7, 8, 9, 10, 11, 12, 14.
Insert 13
13 fits without violating order — placed, no swap.
Insert 5
Placed at the next open slot (child of value 10). 5 < 10 → swap. Continue checking upward.
Key shortcut
Once 5 is confirmed smaller than its direct parent, you don't need to re-check the parent's sibling subtree — if 5 < 10, and 13 > 10 already (heap property), then 5 < 13 is guaranteed transitively. Only the direct ancestor chain needs checking.
Continue up
Compare 5 to its new parent (8) → 5 < 8 → swap. Compare to next parent (7) → 5 < 7 → swap. Now at root, or parent ≥ current → stop.
Why this is only O(log n)
Each step of up-heap does exactly one comparison + at most one swap, and you only ever move up the ancestor chain — never sideways. The ancestor chain length is the tree height, which is log n for a complete tree. So insertion is bounded by O(log n), not O(n).

04 Deletion — Down-Heap

Heaps only support removing the root (the min or max) — that's the whole point of the structure. Steps:

  1. Take the value at the root — this is what gets returned.
  2. Move the last element of the array into the root slot (keeps the tree complete).
  3. Down-heap: compare the new root against its children; if it violates the heap property, swap with the smaller (min heap) / larger (max heap) child, and repeat going downward.

Worked trace

Remove root
Save it (this is the answer to return).
Move last → root
Take the last array element, place it at the root index. Tree stays complete.
Down-heap
Compare current node to left and right children; swap with whichever child breaks the ordering, move down, repeat.
Stop condition
Stop when the node has no more children to check, or it's already correctly ordered relative to both children.

Same logarithmic argument as insertion: at most one comparison-set and one swap per level, and there are log n levels → O(log n) per deletion.

05 Heap Sort

The entire algorithm
While the heap isn't empty: remove the root and append it to a results list.

That's it — no separate "sorting logic" needed. Since every removal returns the current min (or max), repeatedly removing produces a fully sorted sequence for free.

Complexity

StepCost
One removal (down-heap)O(log n)
n removals totalO(n log n)

06 Build Heap (Heapify) from an Unsorted Array — O(n)

Given a raw unsorted array, you can turn it into a valid heap in place — and surprisingly, it costs only O(n), not O(n log n) as a naive analysis might suggest.

The algorithm

Key shortcut
Every leaf node is already a trivially valid 1-node heap — nothing to fix. In a complete binary tree, exactly n/2 of the n nodes are leaves. So skip them entirely, and only run down-heap starting from the last non-leaf node (index ⌊n/2⌋) up to the root. 
for (let i = Math.floor(n / 2); i >= 1; i--) {
  downHeap(i);
}

Why this is O(n), not O(n log n)

A naive count says "n/2 calls × O(log n) each = O(n log n)" — but that overcounts, because most of those calls happen near the bottom of the tree, where down-heap only has a short distance to travel.

Level (from bottom)# of nodes at this levelMax down-heap distance
Leavesn/20 (skipped)
1 above leavesn/41
2 above leavesn/82
.........
root1log n

Summing (nodes at level) × (distance) across all levels gives a series that converges to a small constant multiple of n (a standard result: Σ k / 2^k converges), not n log n. So the total work is bounded by O(n).

Practical takeaway
You don't need to reproduce the summation proof in an interview — just know the result: building a heap from an unsorted array of n elements is O(n), faster than inserting n elements one-by-one (which would cost O(n log n)).

07 Priority Queue

A priority queue is exactly what a heap gives you for free: "give me the highest-priority item first." Min heap → priority = smallest value; max heap → priority = largest value. There's no separate algorithm to learn here — it's the same insert/extract operations, just relabeled by intent.

Alternative: priority queue via linked list

The lecture also walks through maintaining a sorted linked list (smallest at the head) as an alternative:

ImplementationInsertGet highest priority
Sorted linked listO(n) — must traverse to find insertion pointO(1) — it's the head
HeapO(log n)O(1)

The linked-list version trades a slow insert for a fast read — same imbalance problem as the "always keep the array sorted" approach from Section 1. The heap is preferred because it keeps both operations efficient.

08 Java Code (from the lecture)

The lecture builds a generic Heap<T extends Comparable<T>> class backed by an ArrayList. Reconstructed here directly from the walkthrough.

class Heap<T extends Comparable<T>> {
  private ArrayList<T> list = new ArrayList<>();

  private void swap(int first, int second) {
    T temp = list.get(first);
    list.set(first, list.get(second));
    list.set(second, temp);
  }

  private int parent(int i) { return i / 2; }
  private int left(int i)   { return 2 * i; }
  private int right(int i)  { return 2 * i + 1; }

  // insert: add to end, then up-heap
  public void insert(T value) {
    list.add(value);
    upHeap(list.size() - 1);
  }

  private void upHeap(int index) {
    if (index == 0) return;
    int p = parent(index);
    if (list.get(index).compareTo(list.get(p)) < 0) {
      swap(index, p);
      upHeap(p);
    }
  }

  // remove: root goes out, last element becomes new root, then down-heap
  public T remove() {
    if (list.isEmpty()) throw new RuntimeException("Heap is empty");
    T top = list.get(0);
    T last = list.remove(list.size() - 1);
    if (!list.isEmpty()) {
      list.set(0, last);
      downHeap(0);
    }
    return top;
  }

  private void downHeap(int index) {
    int min = index;
    int l = left(index), r = right(index);

    if (l < list.size() && list.get(l).compareTo(list.get(min)) < 0) min = l;
    if (r < list.size() && list.get(r).compareTo(list.get(min)) < 0) min = r;

    if (min != index) {
      swap(index, min);
      downHeap(min);
    }
  }

  // heap sort: drain the heap
  public ArrayList<T> heapSort() {
    ArrayList<T> data = new ArrayList<>();
    while (!list.isEmpty()) data.add(remove());
    return data;
  }
}
Note on indices
This version is written 0-indexed (matching how the code was actually typed in the lecture), even though the theory walkthrough used 1-indexed formulas on the whiteboard — worth double-checking against your own run when you implement it.

09 Complexity Cheat Sheet

Naive: insert n items one-by-oneO(n log n)—
Don't confuse these two
"Build heap from an array you already have" = O(n) (heapify, bottom-up). "Insert n items one at a time into a growing heap" = O(n log n). Same end result, different cost — heapify is strictly better when you have all the data upfront.

10 Mistakes & Interview Tips

❌ Assuming the whole array is sorted
A heap only guarantees the root is correct and that every parent≥children (or ≤). Sibling order and deeper relationships are not guaranteed — don't index into the array expecting sorted order anywhere except the root.
❌ Using O(n log n) insert-by-insert when you have all data upfront
If you're given the full array at once, heapify it in O(n) rather than looping and calling insert() n times (O(n log n)).
❌ Re-checking subtrees you don't need to
During up-heap, once a node is confirmed smaller (min heap) than its direct parent, its parent's sibling subtree doesn't need re-checking — the heap property already guarantees it. Only the direct ancestor chain matters.
💡 Framing for interviews
Heaps exist to answer: "give me the top-priority item, repeatedly, as data streams in — as cheaply as possible for both read and write." That one sentence explains why heaps show up constantly in Top-K, scheduling, and merge-style problems.

Bottom line
Same data structure, same core rules (complete tree + parent/child ordering + O(log n) insert/remove) — the differences are cosmetic (indexing, naming) plus a difference in scope: Kunal's lecture goes deeper into heapify and heap sort, while Alvin's goes deeper into the Top-K interview pattern. Treat them as complementary, not conflicting.
