1. Why Binary Search Is Logarithmic — Understanding Logs

Before the algorithm itself, this course pauses to rebuild the math intuition, since binary search's whole selling point is its runtime — and that runtime is described with a logarithm.

Exponents, in reverse

An exponent like 2⁴ means "multiply 2 by itself 4 times" → 2 × 2 × 2 × 2 = 16. A logarithm asks the reverse question: "2 raised to what power gives me 16?" The answer is 4, so log₂(16) = 4.

The more useful mental model: repeated division

Instead of thinking in exponents, think of a log as counting how many times you can divide a number by the base before it drops to 1 or less:

StepValue
Start16
÷28
÷24
÷22
÷21

Four divisions were needed → log₂(16) = 4. Same idea for log₂(128) = 7 (128 → 64 → 32 → 16 → 8 → 4 → 2 → 1, seven halvings).

Logs can have other bases too (e.g. log₃(243) = 5, since 3⁵ = 243) — but in programming and CS, base 2 is overwhelmingly the default. When you see log(n) written with no base at all, assume base 2.

Why this matters for Big-O

log(n) is always smaller than n — often dramatically smaller:

nlog₂(n)
1,02410
2,04811
The payoff: O(n) (linear) is already a solid complexity — but O(log n) is sublinear, meaning it grows slower than linear and is faster still. Binary search is the textbook example of an algorithm that achieves this by repeatedly halving its search space — which is exactly the "repeated division" pattern a logarithm describes.

2. Classic Binary Search — Approach

Problem: given a sorted array and a target value, return the index where the target is located, or -1 if it isn't present.

Why not brute force? Scanning every position is O(n) — correct, but not taking advantage of the fact that the array is sorted. Binary search exploits that sorted order to achieve O(log n).

The two-pointer setup

  • low starts at index 0 (the beginning).
  • high starts at the last index (the end).
  • Together, [low, high] defines the current "logical subarray" still being searched.

The core loop

  1. Compute mid as the average of low and high, rounded down.
  2. Compare the target against arr[mid]:
    • Target greater than arr[mid] → narrow right: low = mid + 1.
    • Target less than arr[mid] → narrow left: high = mid - 1.
    • Target equal to arr[mid] → found it, return mid.
  3. Repeat until the target is found, or low and high cross (low > high) — at which point the search space is exhausted and the target isn't present.
Worked example (target found): searching for 24 — first mid lands on 27; since 24 < 27, move high left. Next mid lands on 18; since 24 > 18, move low right. Now low and high point at the same element — that's still a valid, size-one search area. Compute mid once more: it matches the target, so its index is returned.
Worked example (target absent): searching for 50 in an array that doesn't contain it — the algorithm keeps narrowing (low creeping right each time arr[mid] < 50) until low and high finally cross (high gets set to mid - 1, landing below the current low). Once low > high, the loop condition while (low <= high) fails, and the algorithm concludes the target was never in the array — return -1.
Why is this O(log n)? Each comparison discards roughly half of the remaining search space, regardless of which direction it goes. Starting from a full array of size n, repeatedly halving down to a single element is precisely what a logarithm counts — the number of times you can divide n by 2 before reaching 1.

Space complexity: O(1) — just the three index variables low, high, and mid, regardless of array size.

3. Classic Binary Search — Implementation

Assumes the array is sorted in increasing order:

function binarySearch(nums, target) {
  let low = 0;
  let high = nums.length - 1;

  while (low <= high) {
    const mid = Math.floor((low + high) / 2);

    if (target < nums[mid]) {
      high = mid - 1;   // search left
    } else if (target > nums[mid]) {
      low = mid + 1;    // search right
    } else {
      return mid;          // target === nums[mid] — found it
    }
  }

  return -1; // low and high crossed — target not present
}
The #1 mistake students make (per the walkthrough): when narrowing the search area, always exclude the just-checked mid from the next range — use mid - 1 / mid + 1, never plain mid. Re-including mid in the next iteration wastes a comparison at best, and can cause an infinite loop at worst (if low/high never actually move).

Time: O(log n). Space: O(1).

4. Binary Search Index (Insertion Point) — Approach

Problem: given a sorted, increasing array and a target, return the index of the target if it's present. If it's not present, return the index where it would need to be inserted to keep the array sorted.

Example: target 25 isn't in the array, but inserting it at index 4 would preserve sorted order — so the answer is 4, not -1.

This is a direct variant of classic binary search — every step of the loop (compute mid, compare, narrow low/high) is identical. The only change is what you do when the loop ends without finding a match.

Why the final value of low is the answer

Walking through a search for a missing target: each time target > arr[mid], low moves to mid + 1; each time target < arr[mid], high moves to mid - 1. The loop keeps running — even when low === high, that's still a valid single-element search area — until the pointers finally cross (low ends up one position past high).

At the moment they cross, low is sitting exactly at the position where the target would need to be inserted to keep every element in order — everything before it is smaller, everything from that position onward (including whatever used to be at high) is larger.

Worked example: searching for 25 — mid sequence lands on 36 (25 < 36 → search left), then 23 (25 > 23 → search right), then 28 (25 < 28 → search left), then 24 (25 > 24 → search right, low = mid + 1). At this point low and high cross. The final value left in low is exactly index 4 — the correct insertion point.

Complexity: identical to classic binary search — O(log n) time, O(1) space.

5. Binary Search Index — Implementation

The entire diff from classic binary search is the final return statement:

function searchInsertIndex(nums, target) {
  let low = 0;
  let high = nums.length - 1;

  while (low <= high) {
    const mid = Math.floor((low + high) / 2);

    if (target < nums[mid]) {
      high = mid - 1;
    } else if (target > nums[mid]) {
      low = mid + 1;
    } else {
      return mid; // target found — same as classic binary search
    }
  }

  return low; // target not found — `low` is the correct insertion index
}

This is the same underlying idea as the "Ceiling of a Number" pattern covered elsewhere in your notes — low/start naturally lands on the smallest element ≥ target once the loop's search space collapses. Different course, same core mechanic.

6. Key Takeaways

  • Logs = repeated halving. log₂(n) counts how many times n can be divided by 2 before reaching 1 — which is exactly what binary search does to its search space on every iteration.
  • O(log n) is sublinear — smaller and faster than O(n), and the gap widens dramatically as n grows.
  • Classic binary search: low = 0, high = length - 1, loop while (low <= high), compute mid, compare, narrow. Never re-include mid in the next range — always mid ± 1.
  • The loop condition low <= high (not <) matters: it ensures a single remaining element still gets checked before the pointers are considered "crossed."
  • Insertion Point variant: identical loop logic; the only change is returning low (instead of -1) when the target isn't found — low always lands exactly where the target would need to be inserted to preserve sorted order.
  • Both variants: O(log n) time, O(1) space — only a constant number of index variables are ever used.
