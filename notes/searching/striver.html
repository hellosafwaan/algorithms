1. Introduction — Binary Search as Real-Life Search

The dictionary analogy: a dictionary is a "search space" that's sorted (words run A → Z). To find "Raj," linear search would flip page by page from the start — technically correct, but slow. Instead, you naturally split the dictionary in half: if the left half contains words starting with "S," everything on the right is "S" or later, and everything with "Raj" must be on the left — so the right half gets discarded entirely, and you repeat the split on the remaining half.

The generalized rule: binary search applies any time you're searching within a sorted search space — and "search space" doesn't have to be an array. It can be a range of possible answers, a set of pages, anything sorted.

2. The Coding Example — Core Mechanics

Array: [3, 4, 6, 7, 9, 12, 16, 17] (indices 0–7). Two pointers define the current search space:

  • low starts at index 0
  • high starts at index n - 1
  • Everything between low and high (inclusive) is the current search space.

Target = 6: mid = (0+7)/2 = 3 → arr[3] = 7. Since 7 ≠ 6, and 6 < 7, discard the right (high = mid - 1 = 2). New mid = (0+2)/2 = 1 → arr[1] = 4. Since 6 > 4, discard the left (low = mid + 1 = 2). Now low === high === 2 → mid = 2 → arr[2] = 6. Match — return index 2.

Target = 13 (not present): the search keeps narrowing — each comparison either bumps low past mid or drops high below mid — until high ends up to the left of low. Once low > high, there's no search space left — return -1.

Watch this exact moment carefully: when the target isn't found, high crosses over to sit just left of low. This "crossing" is the signal the whole loop condition (low <= high) is built around, and it comes back constantly in later problems (lower bound, first/last occurrence, insertion point).

3. Iterative Implementation

function binarySearch(arr, target) {
  let low = 0;
  let high = arr.length - 1;

  while (low <= high) {
    const mid = Math.floor((low + high) / 2);

    if (arr[mid] === target) {
      return mid;
    } else if (target > arr[mid]) {
      low = mid + 1;
    } else {
      high = mid - 1;
    }
  }
  return -1;
}

4. Recursive Implementation + Dry Run

Why recursion fits here: every iteration does the exact same three steps — find mid, compare, narrow — just with a different [low, high] window. Repeating identical steps with shrinking parameters is exactly what recursion is for.

function binarySearchRecursive(arr, low, high, target) {
  if (low > high) return -1; // base case: search space exhausted

  const mid = Math.floor((low + high) / 2);

  if (arr[mid] === target) return mid;
  if (target > arr[mid]) return binarySearchRecursive(arr, mid + 1, high, target);
  return binarySearchRecursive(arr, low, mid - 1, target);
}

Dry run for target 13 (array as above, indices 0–7): call (0,7) → mid=3, arr[3]=7, 13>7 → call (4,7) → mid=5, arr[5]=12, 13>12 → call (6,7) → mid=6, arr[6]=16, 13<16 → call (6,5) → base case hit (low > high) → returns -1, which bubbles back up through every call frame unchanged, all the way to the original caller.

This is the general shape of "recursion unwinding": once the base case returns a value, every enclosing call simply passes that same value straight back up — none of them do extra work on the way out.

5. Time Complexity — The Log Proof

Take an array of size 32. Each comparison eliminates roughly half the remaining elements — sometimes slightly more, sometimes slightly less than exactly half (depending on whether mid itself gets included in the eliminated side), but always approximately half: 32 → 16 → 8 → 4 → 2 → 1, six steps.

32 = 2⁵, so log₂(32) = 5 — close to the six comparisons actually made (the extra step accounts for the final check). Generalizing: for an array of size n, the number of steps to shrink down to 1 element is log₂(n) — because n = 2^k ⟹ log₂(n) = k, and each step is a constant-time comparison.

Time: O(log n). Space (iterative): O(1) — only low, high, mid are used, regardless of array size.

6. The Overflow Case

This matters when the "search space" isn't a small array but something like [0, INT_MAX] — relevant once binary search is applied to answer ranges rather than array indices (see the N-th Root section later).

If the search space has shrunk down to just its very last element (low = high = INT_MAX), then mid = (low + high) / 2 computes INT_MAX + INT_MAX first — which overflows a 32-bit integer before the division even happens.

FixHow
Use a wider typeDeclare low/high/mid as long long (C++) so the sum can't overflow.
Reformulate the midpoint formulamid = low + (high - low) / 2 — algebraically identical to (low + high) / 2, but never sums two large numbers together.
JavaScript's numeric range makes this a non-issue in practice, but the reformulated low + Math.floor((high - low) / 2) is worth using as a habit — it's the version expected in interviews regardless of language.

7. Lower Bound

Definition: the smallest index whose value is ≥ a given number x. If no such value exists, the answer is the array's length (a "hypothetical" index past the end).

Naive: linear scan, O(n).

The answer variable pattern

Unlike classic binary search's three-way branch (equal / greater / less), lower bound only has two real outcomes at each mid: "this could be my answer" (value ≥ x) or "this can't be" (value < x). Track the best candidate seen so far in a separate answer variable, initialized to n (the fallback if nothing qualifies):

function lowerBound(arr, x) {
  let low = 0;
  let high = arr.length - 1;
  let answer = arr.length;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] >= x) {
      answer = mid;       // possible answer — but keep looking left for a smaller index
      high = mid - 1;
    } else {
      low = mid + 1;  // definitely not the answer — go right
    }
  }
  return answer;
}
Since binary search always eliminates the side it moves away from, and lower bound always moves left on a valid candidate, whatever gets stored in answer only ever gets replaced by something smaller. That's what guarantees the final stored value is the smallest valid index — no need to separately compare "is this better than my last answer."

C++ note: the STL provides this built-in — lower_bound(array, array + n, x) returns an iterator; subtract the begin iterator to get the index. No equivalent standard function in Java per the lecture, so the manual implementation is needed there.

Time: O(log n). Space: O(1).

8. Upper Bound

Definition: the smallest index whose value is strictly greater than x (no "or equal to"). Everything else is identical to lower bound — same answer pattern, same loop shape:

function upperBound(arr, x) {
  let low = 0;
  let high = arr.length - 1;
  let answer = arr.length;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] > x) {         // only the comparison operator changes
      answer = mid;
      high = mid - 1;
    } else {
      low = mid + 1;
    }
  }
  return answer;
}

C++ STL equivalent: upper_bound(...). Time: O(log n), Space: O(1) — identical to lower bound.

9. Search Insert Position

Problem: given a sorted array of distinct values and a target, return its index if present — otherwise return the index where it would need to be inserted to keep the array sorted.

This is lower bound, verbatim. If the value exists, lower bound's "smallest index ≥ x" lands exactly on it. If it doesn't exist, lower bound's fallback position is exactly the correct insertion point. No new algorithm needed — just call lowerBound(arr, target).

10. Floor and Ceil in a Sorted Array

Ceil(x): smallest number in the array ≥ x → exactly the value at the lower-bound index (not the index itself).

Floor(x): largest number in the array ≤ x → the mirror image, found with the reversed comparison:

function findFloor(arr, x) {
  let low = 0;
  let high = arr.length - 1;
  let answer = -1; // no floor exists by default

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] <= x) {
      answer = arr[mid];   // store the VALUE, not the index
      low = mid + 1;  // keep looking right for something even closer to x
    } else {
      high = mid - 1;
    }
  }
  return answer;
}
Why no "is this bigger than my last answer" check is needed: since floor always moves right after a valid candidate, and the array is sorted ascending, every subsequent candidate found is guaranteed to be larger than the last. The direction of movement itself enforces the "keep the biggest" invariant — same logic mirrored from lower bound, just flipped.

Read the problem statement carefully — some variants want the returned index, others the returned value. This implementation returns the value; swap answer = arr[mid] for answer = mid if an index is required.

11. First and Last Occurrence of an Element

Naive: linear scan tracking the first and last index where arr[i] === target. O(n).

Approach A — reuse lower bound + upper bound

First occurrence = lowerBound(arr, x), provided that index actually contains x (it might not, if x isn't in the array at all). Last occurrence = upperBound(arr, x) - 1, for the same reason — upper bound finds the first index past all occurrences of x, so stepping back one lands on the last actual occurrence.

function firstAndLast(arr, x) {
  const lb = lowerBound(arr, x);
  if (lb === arr.length || arr[lb] !== x) return [-1, -1]; // x isn't in the array at all
  return [lb, upperBound(arr, x) - 1];
}
Edge case that trips people up: always check the lower-bound result first. If it points past the end of the array, or the value there doesn't match x, x simply isn't present — return [-1, -1] immediately rather than also computing the upper bound (which wastes a second log n pass and, more importantly, would be meaningless).

Approach B — two independent binary searches, no helper functions

Some interviewers specifically want to see binary search written from scratch rather than composed from lower/upper bound helpers. The idea: run binary search twice, biasing the direction after a match each time.

function findFirst(arr, x) {
  let low = 0, high = arr.length - 1, first = -1;
  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] === x) { first = mid; high = mid - 1; } // keep searching LEFT for an earlier one
    else if (arr[mid] < x) low = mid + 1;
    else high = mid - 1;
  }
  return first;
}

function findLast(arr, x) {
  let low = 0, high = arr.length - 1, last = -1;
  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] === x) { last = mid; low = mid + 1; }  // keep searching RIGHT for a later one
    else if (arr[mid] < x) low = mid + 1;
    else high = mid - 1;
  }
  return last;
}

Time: two binary searches back to back is 2 · O(log n) = O(log n) (constants dropped). Space: O(1).

Always call findFirst before findLast, and short-circuit: if findFirst returns -1, return [-1, -1] immediately without running findLast at all — if the element isn't present, there's no last occurrence to find either, and skipping the second search saves a full log n pass.

12. Count Occurrences of a Number in a Sorted Array

Direct extension of the previous problem: with duplicates guaranteed grouped together (sorted array), the count is simply:

function countOccurrences(arr, x) {
  const [first, last] = firstAndLast(arr, x);
  if (first === -1) return 0;
  return last - first + 1;
}

Time: O(log n). Space: O(1).

13. Search in a Rotated Sorted Array (Unique Elements)

A sorted array rotated at some pivot — e.g. [1,2,3,4,5] rotated at index 2 becomes [4,5,1,2,3]. Elements are distinct. Find a target's index.

Why comparing just one side isn't enough

In plain binary search, checking whether the target is less than arr[mid] alone was enough to know which half to eliminate, because both halves were sorted. In a rotated array only one half is guaranteed sorted at any point — the other might straddle the rotation point. So the first, mandatory step every time is: identify which half is sorted.

How to identify the sorted half: compare arr[low] with arr[mid]. If arr[low] <= arr[mid], the left half is sorted (no rotation break in it). Otherwise, the right half must be the sorted one — it's guaranteed that at least one half is always fully sorted, since the rotation point can only fall in one of them.

Once the sorted half is known, check whether the target falls within its value range. If yes, search that half (eliminate the other). If not, the target must be in the other half — eliminate the sorted one instead.

function searchRotated(arr, target) {
  let low = 0;
  let high = arr.length - 1;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] === target) return mid;

    if (arr[low] <= arr[mid]) { // left half is sorted
      if (arr[low] <= target && target < arr[mid]) {
        high = mid - 1;  // target is inside the sorted left half
      } else {
        low = mid + 1;   // target must be on the right
      }
    } else {                    // right half is sorted
      if (arr[mid] < target && target <= arr[high]) {
        low = mid + 1;   // target is inside the sorted right half
      } else {
        high = mid - 1;  // target must be on the left
      }
    }
  }
  return -1;
}

Time: O(log n) — still a single binary search, just with an extra classification step per iteration. Space: O(1).

General principle stated explicitly in the lecture: elimination is the core skill. Once you can reliably identify which chunk of the search space is safe to discard, binary search "writes itself" for almost any variant.

14. Search in a Rotated Sorted Array II (Duplicates)

Same problem, but duplicates are now allowed. You only need to answer whether the target exists (true/false) — not its index, since with duplicates the index wouldn't be unique anyway.

Why the unique-elements approach breaks

The sorted-half identification relied on comparing arr[low] and arr[mid]. With duplicates, a case like [3,3,3,1,3,3,3,3] — searching for 1 — can land low, mid, and high all on the same value (3). At that point the comparison arr[low] <= arr[mid] is uselessly true regardless of which side is actually rotated — there's no information left to classify either half as sorted.

The fix: when arr[low] === arr[mid] === arr[high], this ambiguous case can't be resolved by comparison — so don't try. Just shrink the search space by one from each end (low++, high--) and retry. This never loses the answer, since only known-duplicate boundary values are being trimmed.
function searchRotatedWithDuplicates(arr, target) {
  let low = 0;
  let high = arr.length - 1;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    if (arr[mid] === target) return true;

    if (arr[low] === arr[mid] && arr[mid] === arr[high]) {
      low++;
      high--;
      continue; // re-evaluate from scratch with the shrunk space
    }

    if (arr[low] <= arr[mid]) {
      if (arr[low] <= target && target < arr[mid]) high = mid - 1;
      else low = mid + 1;
    } else {
      if (arr[mid] < target && target <= arr[high]) low = mid + 1;
      else high = mid - 1;
    }
  }
  return false;
}

Complexity: average case is still O(log n). Worst case (e.g. an array of near-uniform duplicates like [3,3,1,3,3,3,3]) degrades to O(n/2) ≈ O(n), since the ambiguous case can only shrink the space by one element at a time instead of halving it. Space: O(1).

A generalizable strategy for "with duplicates" follow-ups: solve the unique-elements version first, then look specifically for the test case where your classification comparison becomes ambiguous due to duplicates, and patch exactly that case. This pattern (solve unique → find the duplicate-breaking case → patch it) reapplies directly to Find Minimum in a Rotated Sorted Array as well.

15. Find Minimum in a Rotated Sorted Array

Same rotated-array setup; this time return the minimum value, not an index, and there's no target to search for.

Core insight

At every step, identify the sorted half (same technique as Q13). The sorted half's minimum is always its leftmost element — so grab that as a candidate answer, then discard the entire sorted half (it's already been fully accounted for) and keep searching the unsorted half, which is where the true rotation point — and therefore the true minimum — must still be hiding.

function findMin(arr) {
  let low = 0;
  let high = arr.length - 1;
  let answer = Infinity;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);

    // optimization: if this whole remaining chunk is already sorted, its
    // leftmost element IS the minimum — no need to keep searching further
    if (arr[low] <= arr[high]) {
      answer = Math.min(answer, arr[low]);
      break;
    }

    if (arr[low] <= arr[mid]) {       // left half is sorted
      answer = Math.min(answer, arr[low]);
      low = mid + 1;                 // eliminate the (already-accounted-for) sorted left half
    } else {                            // right half is sorted
      answer = Math.min(answer, arr[mid]);
      high = mid - 1;                // eliminate the sorted right half
    }
  }
  return answer;
}
Why the "already fully sorted" optimization is safe: when arr[low] <= arr[high] for the entire current window, it means the rotation point isn't inside this window at all — the whole remaining chunk is one contiguous sorted run, so its first element is trivially the minimum, and further binary search steps would be redundant work.

Time: O(log n) for unique elements. Space: O(1). (Duplicates: apply the same shrink-by-one patch from Q14 as a follow-up exercise.)

16. Find How Many Times an Array Has Been Rotated

Key realization: the number of rotations equals the index at which the minimum element sits. If the minimum is at index 3, the array has been rotated exactly 3 times.

function countRotations(arr) {
  let low = 0;
  let high = arr.length - 1;
  let minValue = Infinity;
  let minIndex = 0;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);

    if (arr[low] <= arr[high]) {
      if (arr[low] < minValue) { minValue = arr[low]; minIndex = low; }
      break;
    }

    if (arr[low] <= arr[mid]) {
      if (arr[low] < minValue) { minValue = arr[low]; minIndex = low; }
      low = mid + 1;
    } else {
      if (arr[mid] < minValue) { minValue = arr[mid]; minIndex = mid; }
      high = mid - 1;
    }
  }
  return minIndex;
}

This is a two-line diff from Find Minimum — just track the index alongside the value whenever answer gets updated, and return the index instead of the value.

17. Single Element in a Sorted Array

Every number appears exactly twice except one, which appears once — find it. Array is sorted, so duplicate pairs sit adjacent.

Naive: for each index, check whether its left or right neighbor matches (careful with the array boundaries at index 0 and n-1) — O(n).

The even/odd index pattern

Before the single element, every pair starts at an even index (0,1 then 2,3 then 4,5...). After the single element throws things off by one, every pair instead starts at an odd index. That shift is the signal:

  • At an even index: if arr[i] === arr[i+1], you're still in the "before" region — the single element is to the right.
  • At an odd index: if arr[i] === arr[i-1], you're already in the "after" region — the single element is to the left.

Avoiding edge-case boilerplate

Handle the size-1 array, and the first/last elements, as separate explicit checks up front — then run binary search only on the trimmed range [1, n-2], so mid ± 1 inside the loop can never go out of bounds.

function singleNonDuplicate(arr) {
  const n = arr.length;
  if (n === 1) return arr[0];
  if (arr[0] !== arr[1]) return arr[0];
  if (arr[n - 1] !== arr[n - 2]) return arr[n - 1];

  let low = 1;
  let high = n - 2;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);

    if (arr[mid] !== arr[mid + 1] && arr[mid] !== arr[mid - 1]) {
      return arr[mid]; // matches neither neighbor — this is the single element
    }

    const isEvenIndex = mid % 2 === 0;
    const onLeftHalf = (isEvenIndex && arr[mid] === arr[mid + 1]) ||
                       (!isEvenIndex && arr[mid] === arr[mid - 1]);

    if (onLeftHalf) {
      low = mid + 1; // still in the "before" region — single element is to the right
    } else {
      high = mid - 1; // already past it — single element is to the left
    }
  }
  return -1; // unreachable given problem guarantees; satisfies return type
}

Time: O(log n). Space: O(1).

18. Find Peak Element

Definition: arr[i] is a peak if it's strictly greater than both neighbors. The array's boundaries are treated as bordered by -Infinity on both sides — so the very first or last element can qualify as a peak by being greater than just its one real neighbor.

The array may have multiple peaks — any one of them is an acceptable answer.

Visualize it as a curve: think of the array as alternating increasing/decreasing runs. If arr[mid] > arr[mid+1], mid sits on a decreasing slope, which guarantees a peak exists somewhere to the left (including possibly at mid itself). If arr[mid] < arr[mid+1], mid is on an increasing slope, guaranteeing a peak to the right. Either way, eliminating the opposite half never eliminates a guaranteed peak.

Handling the boundaries cleanly

Check the size-1 case and the "peak at index 0" / "peak at index n-1" cases explicitly up front, then run binary search only on the interior range [1, n-2] — avoiding any need to special-case mid - 1 < 0 or mid + 1 >= n inside the loop.

function findPeakElement(arr) {
  const n = arr.length;
  if (n === 1) return 0;
  if (arr[0] > arr[1]) return 0;
  if (arr[n - 1] > arr[n - 2]) return n - 1;

  let low = 1;
  let high = n - 2;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);

    if (arr[mid] > arr[mid - 1] && arr[mid] > arr[mid + 1]) {
      return mid; // found a peak
    } else if (arr[mid] > arr[mid - 1]) {
      low = mid + 1;   // increasing slope — a peak is guaranteed to the right
    } else {
      high = mid - 1;  // decreasing slope (or a "valley" case) — a peak is guaranteed to the left
    }
  }
  return -1; // unreachable given the problem's guarantees
}
The single-peak-only bug: an earlier, more "careful"-looking version explicitly checked three cases — "is a peak," "on increasing slope," "on decreasing slope" — with no else fallback. That works when the array has exactly one peak, but breaks (infinite loop) on arrays with multiple peaks, because a local valley point (smaller than both neighbors) satisfies none of the three explicit conditions. The fix is using a plain if / else if / else — whenever mid isn't a peak and isn't strictly on an increasing slope, it's always safe to go left, because a peak is guaranteed on both sides whenever you're discarding one — the eliminated side is never the only side that could hold a peak.

Time: O(log n). Space: O(1).

19. N-th Root of a Number (Binary Search on the Answer)

Problem: given integers n and m, find x such that xⁿ = m. If no such integer exists, return -1. (Finding a square root is just the special case n = 2.)

Naive: try every integer from 1 upward, computing iⁿ each time, until it matches m, exceeds it (impossible from here on — stop and return -1), or the loop runs out. O(m), times whatever the exponentiation itself costs.

Why this is binary-search-shaped despite having no array: as the candidate x increases, xⁿ increases monotonically. That monotonic relationship — small guesses undershoot, large guesses overshoot, and the "correct" zone is somewhere continuous in between — is exactly what binary search needs. The "search space" here is the range of possible answer values, from 1 to m, not an array.
function nthRoot(n, m) {
  let low = 1;
  let high = m;

  while (low <= high) {
    const mid = low + Math.floor((high - low) / 2);
    const cmp = compareToPowerN(mid, n, m);

    if (cmp === 0) return mid;         // mid^n === m exactly
    else if (cmp < 0) low = mid + 1;   // mid^n < m — try a bigger candidate
    else high = mid - 1;                // mid^n > m — try a smaller candidate
  }
  return -1; // no exact n-th root exists
}

The overflow trap — and why the fix isn't "just use a bigger int type"

Naively computing mid ** n directly can explode enormously — e.g. finding the 10th root of 10⁹ puts mid around 5×10⁸, and raising that to the 10th power lands around 10⁹⁰, which overflows even 64-bit integers by a wide margin. Switching to a larger numeric type doesn't actually solve this — the number itself is just too big to represent.

The real fix: don't compute the full power at all. Multiply mid by itself one factor at a time, and bail out the moment the running product exceeds m — at that point you already know the comparison result and don't need to keep multiplying toward an astronomically large, useless final number.
// returns -1 if mid^n < m, 0 if mid^n === m, 1 if mid^n > m — without ever computing the full power
function compareToPowerN(mid, n, m) {
  let product = 1;
  for (let i = 1; i <= n; i++) {
    product *= mid;
    if (product > m) return 1; // already exceeded — stop early, no need to keep multiplying
  }
  if (product === m) return 0;
  return -1;
}

Complexity: the outer binary search runs O(log m) times; each call to the comparison helper costs up to O(n) multiplications. Total: O(n · log m). Space: O(1).

The lecture also mentions a faster binary-exponentiation method for computing powers in O(log n) instead of O(n) — useful, but the early-exit multiplication loop above is preferred here specifically because it sidesteps the overflow problem, which binary exponentiation alone does not.

21. Key Takeaways

  • The one skill underlying everything: at each step, identify exactly which chunk of the search space is safe to eliminate. Every variant in this series — lower/upper bound, rotated arrays, peak finding, binary search on answers — reduces to this.
  • The answer variable pattern: whenever a problem isn't a clean "find the exact match" search (lower bound, upper bound, floor, ceil), track the best candidate seen so far in a separate variable rather than trying to force it into classic binary search's three-way branch.
  • Rotated arrays: always identify the sorted half first (compare arr[low] vs arr[mid]) before deciding where the target/minimum could be. Duplicates break this comparison when arr[low] === arr[mid] === arr[high] — shrink by one from each end in that case.
  • Boundary-heavy problems (Single Element, Peak Element): handle index-0 and index-(n-1) as explicit special cases up front, then binary search only the safe interior range — this avoids scattering mid ± 1 bounds-checks throughout the main loop.
  • Binary search on the answer: applicable whenever candidate answers have a monotonic relationship to a check function (bigger guess → check gets easier or harder in one consistent direction) — even with no array in sight, as in N-th Root.
  • Overflow: use mid = low + Math.floor((high - low) / 2) instead of Math.floor((low + high) / 2), and when comparing computed values against a target (e.g. a power), bail out early the moment a running computation exceeds the target rather than computing the full (potentially enormous) result.
