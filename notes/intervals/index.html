Part A · Foundations

What is an interval, really?

Before any LeetCode problem — the mental model, the formulas, and why sorting by start time is almost always step one.

01Building the intuition

Two people are trying to find overlapping free time. Person A is free 3:00–8:00. Person B is free 4:00–9:00. Visually, their shared window is 4:00–8:00 — that shared window is an interval, and finding it is the core operation this whole pattern is built around.

Formal definition. Many real-world problems reduce to ranges: time ranges, numeric ranges, memory ranges, geometric ranges. Whenever the input looks like an array of [start, end] pairs, think "interval pattern." An interval by itself isn't interesting — it becomes useful once compared against another interval. That comparison is the whole pattern.

03When do two intervals not overlap?

It's easier to reason about non-overlap first, then negate it. Given I1 = [s1, e1] and I2 = [s2, e2], they do not overlap if either interval starts strictly after the other has already ended:

s2 > e1   OR   s1 > e2
Generic two-interval setup
I1
s1 ⟶ e1
I2
s2 ⟶ e2
overlap
Here e1 ≥ s2 (I2 started before I1 ended) and e2 ≥ s1 (I1 started before I2 ended) — both true, so they overlap.

04Deriving the overlap formula via De Morgan's Law

De Morgan's Law: NOT(A OR B) = (NOT A) AND (NOT B). A point outside both circles below satisfies "not in A" and "not in B" simultaneously — that's the equivalence.

A ∪ B vs. ¬A ∧ ¬B (De Morgan visualized)
Everything outside both circles is where De Morgan's equivalence lives — the region this course used to flip the "definitely non-overlapping" condition into the overlap formula.

Applying De Morgan's Law to the non-overlap condition and negating both sides:

overlapping  =  NOT[(s2 > e1) OR (s1 > e2)]
           =  NOT(s2 > e1)  AND  NOT(s1 > e2)   // De Morgan's Law
           =  (e1 >= s2)  AND  (e2 >= s1)
⭐ The overlap formula
e1 >= s2 && e2 >= s1 — the single most reused formula in interval problems.

06Why sort by start time first?

This is the single most important first step in nearly every interval problem. If intervals arrive unordered, comparing arbitrary pairs to find overlaps/gaps/merges becomes chaotic — you can't reliably tell what's "done."

Counter-example: why NOT end time

[1,10] · [2,3] · [4,5] — the trap
[1,10]
1 – 10
[2,3]
2–3
[4,5]
4–5
0
5
10
Sorted by end time: [2,3], [4,5], [1,10] — [4,5] sits between them in the scan order, so by the time [1,10] is reached, the valid merge with [2,3] has already been "stepped past" and gets missed.

Sorted by start time: [1,10], [2,3], [4,5] — both later bars get correctly absorbed into [1,10] in a clean left-to-right scan.
Takeaway: sorting by start time guarantees once an interval is processed, nothing earlier in order can retroactively need to merge with something later — comparison proceeds strictly left-to-right. End-time sort doesn't give this guarantee.

07When to recognize the interval pattern

Reach for this pattern whenever the input contains time ranges, numeric ranges, memory ranges, or coordinate ranges, and the question asks about overlaps, gaps, ordering, capacity, merging, or total coverage.

Sorting by start time solves the biggest part of the problem in the vast majority of interval questions. (Sorting by end time occasionally matters for a different class — e.g. greedy interval scheduling — but for merge/overlap/gap-style problems, start-time sort is the default.)

08LC 986 — Interval List Intersections

Problem: given two lists of closed intervals, each pairwise disjoint and sorted, return their intersection.

Key insight: because intervals within one list never overlap each other, any intersection can only happen between an interval from A and an interval from B — never A-with-A or B-with-B.

Step 1: A[0]=[1,3] vs B[0]=[2,5] → overlap [2,3] pushed. Since e1(3) < e2(5), advance i.
Step 2: A[1]=[5,7] vs B[0]=[2,5] → overlap [5,5] (the coral dot — a single-point intersection) pushed. Since e1(7) > e2(5), advance j — loop ends.
Pointer-advance rule: regardless of whether the current pair overlapped, always advance whichever interval ends first — by the disjoint+sorted guarantee, it can never contribute a further intersection.
JavaScript
function intervalIntersection(firstList, secondList) {
  const result = [];
  let i = 0, j = 0;

  while (i < firstList.length && j < secondList.length) {
    const [s1, e1] = firstList[i];
    const [s2, e2] = secondList[j];

    const start = Math.max(s1, s2);
    const end = Math.min(e1, e2);
    if (start <= end) result.push([start, end]);

    // advance whichever interval ends first
    if (e1 < e2) i++; else j++;
  }
  return result;
}
ComplexityReasoning
TimeO(m + n)Each step advances i or j by 1
SpaceO(m + n)Worst case: every interval produces its own piece
MistakeWhy it's wrong
Using merge formula instead of intersection formulaMerge and intersection are opposite operations
Advancing by start time instead of end timeThe "exhausted" guarantee is about end times, not starts
Only advancing when an overlap is foundMust advance even on no-overlap — the finishing interval is still "done"

09LC 3169 — Count Days Without Meetings

Problem: given days total available days and a list of (possibly overlapping) meetings, count days with no meeting scheduled.

Builds directly on Merge Intervals: sort → merge overlapping meetings → sum the gaps between merged blocks, plus the gap before the first block and after the last.

No overlaps here, so merge changes nothing. Free days (coral): day 4 and day 8 → gap total = 2.
⚠ The classic gotcha: a naive version only sums the internal gaps between merged blocks and forgets the edges: gap += result[0].start - 1 (before the first meeting) and gap += days - result[last].end (after the last meeting). Missing either gives a too-small answer.
JavaScript
function countDays(days, meetings) {
  meetings.sort((a, b) => a[0] - b[0]);

  const merged = [meetings[0]];
  for (let i = 1; i < meetings.length; i++) {
    const [start, end] = meetings[i];
    const last = merged[merged.length - 1];
    if (start <= last[1]) last[1] = Math.max(last[1], end);
    else merged.push([start, end]);
  }

  let gap = 0;
  for (let i = 1; i < merged.length; i++) {
    gap += merged[i][0] - merged[i - 1][1] - 1;
  }
  gap += merged[0][0] - 1;                          // before first block
  gap += days - merged[merged.length - 1][1];        // after last block
  return gap;
}
ComplexityReasoning
TimeO(n log n)Dominated by the sort; merge + gap-sum is O(n)
SpaceO(n)Worst case, no merges happen

10Brute force: Merge Intervals (pairwise comparison)

Having already solved Merge Intervals with sort + linear-merge (O(n log n)), this explores the brute-force alternative — useful for the interview question "why is your approach better than brute force?"

The idea: compare every pair, merge, repeat

Without sorting, compare every pair of intervals; whenever two overlap, merge them and delete the redundant one. A single full pass isn't enough — merging can grow an interval into overlap range it was never compared against.

Input: [5,7] · [1,4] · [12,15] · [3,5] · [9,11] · [2,6]
Pass 1 [2,7] [1,4] [12,15] [9,11] [3,5] [2,6]
Pass 2 [1,7] [12,15] [9,11] [2,7] [1,4]
Pass 3 [1,7] [12,15] [9,11] ← no merges → done
Pass 1: [5,7] merges with [3,5] → [3,7], then with [2,6] → [2,7]. But [2,7] and [1,4] genuinely overlap and are never compared in this pass — [2,7] formed too late in the scan order.

Pass 2: catches the missed merge → [2,7] + [1,4] → [1,7]. A merge happened, so run again.

Pass 3: zero merges found → algorithm terminates. Final: [[1,7],[12,15],[9,11]] ✓
Pseudocode
list = copy of all intervals
isMergeHappened = true

while isMergeHappened:
    isMergeHappened = false
    i = 0
    while i < list.size:
        j = i + 1
        while j < list.size:
            if isOverlap(list[i], list[j]):
                list[i].start = min(list[i].start, list[j].start)
                list[i].end   = max(list[i].end,   list[j].end)
                list.remove(j)      # do NOT increment j
                isMergeHappened = true
            else:
                j = j + 1
        i = i + 1
return list
JavaScript
function isOverlap(a, b) {
  return a[1] >= b[0] && b[1] >= a[0];
}

function mergeIntervalsBruteForce(intervals) {
  const list = intervals.map(iv => [...iv]);
  let isMergeHappened = true;

  while (isMergeHappened) {
    isMergeHappened = false;
    for (let i = 0; i < list.length; i++) {
      let j = i + 1;
      while (j < list.length) {
        if (isOverlap(list[i], list[j])) {
          list[i][0] = Math.min(list[i][0], list[j][0]);
          list[i][1] = Math.max(list[i][1], list[j][1]);
          list.splice(j, 1);   // remove; do NOT advance j
          isMergeHappened = true;
        } else {
          j++;
        }
      }
    }
  }
  return list;
}

11Consolidated takeaways

  1. Two intervals overlap iff e1 >= s2 && e2 >= s1 — derived via De Morgan's Law from the "definitely non-overlapping" condition.
  2. Merge [min(s), max(e)] vs. Intersection [max(s), min(e)] — opposite operations, easy to swap by mistake.
  3. Always sort by start time first for merge/overlap/gap problems — it converts chaotic pairwise reasoning into a clean left-to-right scan.
  4. Brute-force pairwise comparison needs repeated full passes until a pass produces zero merges — a single pass misses merges formed late in the scan.
  5. When mutating a working copy of an array mid-algorithm, audit every bound check and read for consistent reference to the live structure.
