Plan, stated cleanly and quickly: clone the starting node, then iterate its neighbors and clone each one.

## Where He Got Stuck
Stress-tested his plan against the simplest possible cycle (two nodes, A and B, each other's only neighbor). Correctly predicted that naively re-cloning A when reached again via B would either duplicate work or infinite-loop, and — echoing the LC 200 lesson directly — arrived at "add it before recursing, or we end up in an infinite loop" essentially unprompted.

From there, built the plan piece by piece:
- Needed a `Map` (not a `Set`) to go from original node → its clone, correctly recalling the Set-for-membership/Map-for-key-value distinction from LC 217.
- Needed a direct explanation that `Map` supports object references as keys (unlike plain `{}`, which coerces to strings) — genuine toolkit fact, not derivable from tracing.

## Mistakes Made
All mechanical, all caught within one question or trace each:
- **Constructor called without `new`** (pattern #47) — `_Node(node.val, node.neighbors)` instead of `new _Node(...)`. One direct question, immediate correct answer.
- **Passed the original `node.neighbors` array directly into the clone** (pattern #48) — would have shared the original (un-cloned) neighbor references instead of building fresh clones. Caught once asked directly what would end up inside the result.
- **Loop variable shadowing** (pattern #49) — `for (const node of node.neighbors)`, reusing the outer parameter name. Mechanically works in JS due to scoping order, but a genuine readability trap; flagged directly since it's obscure JS behavior, not something to discover via tracing.
- **Registered the clone in the map AFTER building the neighbor list, not before** (pattern #50) — despite having *just stated* the correct rule verbally minutes earlier. Caught by re-tracing the same A/B cycle against the actual code order, arriving at "infinite loop" himself. Good demonstration that a correct verbal rule doesn't guarantee the code implements it in that order — worth re-verifying via trace even when the stated understanding sounds right.

## Key Insight
"We add it before recursing. If we add it after recursing, we will end up in an infinite loop" — direct transfer of the LC 200 mark-before-recurse principle to a new problem, stated in his own words before writing the code (even though the first draft didn't actually follow it).

Second, bigger insight, arrived at via comparing all three solutions built this session: **DFS vs BFS isn't about recursion vs loops, or the code's shape — it's entirely about which end of the pending-work list you pull from.** The iterative BFS and iterative DFS solutions are identical except for one line (`queue.shift()` vs `stack.pop()`).

## Complexity Reached
Time: O(V + E) — every node is created/registered exactly once (map check prevents re-cloning); every edge is traversed once from each direction it's discovered (bounded by total neighbor-list length across all nodes).
Space: O(V) — the map holds one entry per node; recursion stack (DFS version only) can also reach O(V) in the worst case (long chain).
