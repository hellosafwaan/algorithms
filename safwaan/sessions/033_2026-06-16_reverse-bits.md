# Session: Reverse Bits — 2026-06-16

## What He Attempted
First instinct was to rotate/shift the whole number: "I'm thinking we need to reverse the whole number... move the bits... like some kind of left shift or right shift... if the number consists of 10 bits, I have to move the bits either to the left or to the right 10 times, and then the bits will be reversed." This is rotation, not reversal — confusing the two.

## Where He Got Stuck
1. **Rotate vs reverse** — believed repeated shifting would produce the reversed bit order. Resolved by tracing `1100` rotated right 4 times (its own width) — landed back at the start, exposing that rotation cycles rather than mirrors.
2. **Building `result`** — once the mirror relationship (`position i → position (n-1-i)`) was established, didn't know how to place an extracted bit into a growing result. Walked through OR's identity property (`0` start value) and the need to "make room" before "filling."
3. **OR-then-shift vs shift-then-OR ordering** — proposed OR-ing the bit in first, then left-shifting. This re-shifts the just-placed bit on the same iteration, corrupting position. A verbal explanation didn't land; a full iteration-by-iteration trace table (two orderings, same input `1010`) did. He explicitly confirmed this approach worked and asked for more trace tables going forward.
4. **OR-then-right-shift alternative** — tested whether right-shifting `result` after OR-ing would also work. It doesn't — the bit gets pushed off the edge and discarded immediately. Caught with another trace table.
5. **`while(n !== 0)` terminating early** — used the LC 191 loop shape, but for *building* a positionally-correct result, the loop needs to run a fixed number of times (32) regardless of when `n`'s remaining bits become `0`. Traced `n = 1` and found the loop only ran once, leaving the bit at position 0 of `result` instead of position 31 where it belongs.
6. **`>> ` vs `>>>`** — used the signed right shift for advancing `n`, which would sign-extend (fill with `1`s) for any `n` with the leftmost bit set, causing an infinite loop. Needed direct explanation since this was a genuine knowledge gap.
7. **Why force unsigned output when input is "signed"** — pushed back on `>>> 0` before returning. Good instinct. Resolved by separating the *declared type* (signed int, a labeling convention) from what the algorithm actually does (treats bits as a positionless pattern). Then independently noticed the actual LeetCode constraints (`n` even, `0 <= n <= 2^31-2`) guarantee the result's sign bit is always `0` — so `>>> 0` isn't strictly needed for this constrained version, though it's good defensive practice generally.

## Mistakes Made
- Confused rotation with reversal (corrected via trace)
- OR-then-shift ordering bug (corrected via trace table)
- OR-then-right-shift idea (also fails — caught via trace table)
- `while(n !== 0)` instead of a fixed 32-iteration loop (caught via trace of `n = 1`)
- `n >> 1` instead of `n >>> 1` for advancing `n` (knowledge gap, given directly)

All caught through guided questioning and concrete tracing — no bug slipped into the final submission unaddressed.

## Key Insight
In his own words, after the shift/OR confusion resolved: "Once we have the last digit, we need to append it to the result using OR. But before we do that we have to do a left shift so that we get free space on the right bit for the next number to come."

The deeper insight that took longest to land: reversing means mirroring positions (`i → n-1-i`), not rotating. And building a result bit-by-bit requires making room (shift) before filling (OR) — doing it in the other order corrupts already-placed bits on every subsequent iteration.

## Complexity Reached
Time: O(32) → O(1) — fixed 32 iterations regardless of input value, same reasoning as LC 191's fixed-width collapse.
Space: O(1) — one accumulator variable.
