# Session: Rotate Array — 2026-07-17

## What He Attempted

Wrote a brute-force solution from scratch:

```js
function rotate(nums, k) {
    for(let i = 0; i < k; i++) {
        const element = nums.pop();
        nums.unshift(element);
    };
};
```

Got Time Limit Exceeded and brought it to the session asking why.

## Where He Got Stuck

Two distinct points:
1. The optimization itself — needed guided tracing to arrive at the destination-index formula `(i+k)%n`, and once there, explicitly said the modulo-wraparound concept "does not feel familiar" (despite already having used `%` for digit extraction) — needed full direct teaching with a trace table.
2. The two O(1)-space follow-ups (three reversals, cyclic replacement) — explicitly asked to be taught these directly rather than derived ("teach me like a baby," "don't question me").

## Mistakes Made

- Original brute force: O(n·k) from `k` calls to `unshift` (each O(n)) — correctly self-diagnosed this was the TLE cause once asked about `unshift`'s cost, without needing the answer given.
- First draft of the extra-array solution `return`ed the new array instead of copying it back into `nums` — caught himself when asked to re-check the problem's `void`/in-place requirement.

## Key Insight

Independently and unprompted identified the aliasing problem: writing `nums[(i+k)%n] = nums[i]` directly in place would destroy a value before it's read for its own destination mapping. This is the real reason naive in-place writing doesn't work, and he saw it without being asked — a genuine correctness catch, distinct from the TLE issue.

## Complexity Reached

Brute force: O(n·k) time, O(1) space (TLE). Extra array (submitted): O(n) time, O(n) space. Three reversals / cyclic replacement (taught, not self-derived): O(n) time, O(1) space each.
