# Session: Pow(x, n) — 2026-06-07

## What He Attempted

**Attempt 1 — Iterative (positive n only):**
```js
function myPow(x, n) {
    let result = x
    for (let i = 2; i <= n; i++) {
        result = result * x
    }
    return result
}
```
Handled positive n correctly. Missed n = 0 and negative n. He noted the comment himself.

**Attempt 2 — Iterative with negative n:**
```js
function myPow(x, n) {
    let poweredValue = x;
    let timesToBePoweredBy = Math.abs(n)
    for (let i = 2; i <= timesToBePoweredBy; i++) {
        poweredValue = poweredValue * x
    }
    if(n < 0) return 1 / poweredValue
    else return poweredValue
}
```
Handled negatives. Still missed n = 0 (poweredValue starts at x, loop skips, returns x not 1).

**Attempt 3 — Iterative, handles 0:**
```js
function myPow(x, n) {
    let poweredValue = 1;
    let timesToBePoweredBy = Math.abs(n)
    for (let i = 1; i <= timesToBePoweredBy; i++) {
        poweredValue = poweredValue * x
    }
    if(n < 0) return 1 / poweredValue
    else return poweredValue
}
```
302/307 test cases — timed out on very large n. Logic correct, O(n) complexity the problem.

**Attempt 4 — Recursive, positive n only:**
```js
function myPow(x, n) {
    if (n === 1) return x;
    return x * myPow(x, n - 1)
}
```
No base case for n = 0 → infinite recursion. Caught it himself after testing.

**Attempt 5 — Recursive with n = 0:**
```js
function myPow(x, n) {
    if (n === 0) return 1;
    if (n === 1) return x;
    return x * myPow(x, n - 1)
}
```
Handles positive and zero. Negative n → infinite recursion (decrements forever).

**Attempt 6 — Recursive with negative n:**
```js
function myPow(x, n) {
    if (n === 0) return 1;
    if (n === 1) return x;
    if(n < 0) return (1 / x) * myPow(x, n + 1)
    return x * myPow(x, n - 1)
}
```
Logic correct. Stack overflow on large n — still O(n) depth.

**Attempt 7 — Removed redundant n=1 base case:**
```js
function myPow(x, n) {
    if (n === 0) return 1;
    if(n < 0) return (1 / x) * myPow(x, n + 1)
    return x * myPow(x, n - 1)
}
```
Same problem — O(n) depth. This is where he hit the wall and asked for a nudge.

**Attempt 8 — Exponentiation by squaring (first draft, double-call bug):**
```js
function myPow(x, n) {
    if (n === 0) return 1;
    if(n % 2 === 0) return myPow(x, n/2) * myPow(x, n/2)
    else return x * myPow(x, n/2) * myPow(x, n/2)
}
```
Called myPow(x, n/2) twice — doubles work, defeats O(log n). Caught by question.

**Attempt 9 — Stored half, scope bug:**
Tried to store powerOfHalfed inside if/else blocks — variable inaccessible outside.

**Attempt 10 — Handled negative with -n inside recursion:**
Lost track of structure. Overcomplicated.

**Final solution:**
```js
function myPow(x, n) {
    if (n === 0) return 1;
    if (n < 0) return 1 / myPow(x, -n)
    const powerOfHalfed = myPow(x, Math.floor(n/2))
    if(n % 2 === 0) return powerOfHalfed * powerOfHalfed
    else return x * powerOfHalfed * powerOfHalfed
}
```

## Where He Got Stuck

Two main walls:
1. **After the O(n) recursive approach** — he had exhausted his toolbox and said so explicitly: "I've exhausted my tool list." Needed the exponentiation by squaring concept introduced from scratch. He hadn't seen it before.
2. **n/2 gives a float for odd n** — when he got Infinity for `myPow(2, 10)`, he traced through the call tree on paper and couldn't see it. The question "what does n/2 give you in JavaScript?" unlocked it immediately.

## Mistakes Made

- Missed n = 0 on first two iterative attempts — only discovered after submitting
- Missed negative n on first iterative attempt — only discovered after submitting
- Called `myPow(x, n/2)` twice instead of storing it — double work, kills O(log n)
- Scoped `powerOfHalfed` inside if/else blocks — inaccessible outside
- Used `n/2` without `Math.floor` for odd n — float causes infinite recursion, shows up as Infinity

## Key Insight

Safwaan independently formulated: `x^n = x^i * x^j where i + j = n`. He then reasoned that i and j can be anything — (9,1), (8,2), (4,6) — but i=j=n/2 is optimal because it's the only split where both subproblems are identical, so you only compute one and reuse it. Any other split gives two *different* subproblems.

From that: compute `x^(n/2)` once, store it, square it. Instead of O(n) steps, O(log n). For odd n, pull out one extra `x`. Handle negative n once at the top: `1 / myPow(x, -n)`.

## Complexity Reached

Time: O(log n) — Space: O(log n) (call stack depth)
